VulnOps Systems Architecture

End-to-end vulnerability remediation pipeline design

System Architecture

System Components

1. Scanner API Integration: Python scripts poll Qualys, Nessus, Rapid7, and Tenable APIs every 6 hours. Parse JSON/XML responses, normalize data formats, and store in intermediate database.

2. Enrichment & Attribution: Match vulnerabilities to assets using CMDB data. Leverage internal LLM to cluster similar findings, identify false positives, and generate standardized descriptions. Resolve misattribution using hostname/IP/instance ID mapping.

3. Automated Ticketing: Create ServiceNow/Jira tickets with proper owner assignment based on service ownership mapping. Include severity, CVSS score, affected assets, remediation guidance, and SLA deadline.

4. SLA Governance: Track remediation progress against SLA deadlines. Automated escalation at 7-day warning, 3-day critical, and executive notification for SLA breaches.

5. Verification & Closure: Trigger rescans after remediation. Validate fix effectiveness. Auto-close tickets upon successful verification.

6. Executive Reporting: Tableau dashboards showing vulnerability trends, SLA compliance, remediation velocity, and risk posture across all environments.

Technical Stack

Languages: Python 3.9+, Bash
APIs: Qualys VMDR, Nessus, Rapid7 InsightVM, Tenable.io
Ticketing: ServiceNow REST API, Jira API
Data: PostgreSQL, Redis (caching)
CMDB: ServiceNow CMDB, AWS Config, Azure Resource Graph
LLM: Internal LLM API (scanner output synthesis)
Reporting: Tableau Server, Python (pandas, matplotlib)
Orchestration: Apache Airflow (DAGs for scheduled tasks)
Monitoring: Splunk, Prometheus, Grafana
Infrastructure: Docker, Kubernetes, AWS ECS

Data Flow

Step 1: API Ingestion
Scanner API → JSON Response → Parse & Normalize → Store Raw Data

Step 2: Enrichment
Raw Vuln Data + CMDB Data → Asset Matching → Owner Identification
→ LLM Analysis (clustering, false positive detection) → Enriched Finding

Step 3: Ticket Creation
Enriched Finding → Template Generation → ServiceNow API Call
→ Ticket Created with Owner Assignment → SLA Timer Starts

Step 4: SLA Tracking
Daily Scan of Open Tickets → Calculate Days Remaining
→ Trigger Escalations (7-day, 3-day, breach) → Notify Stakeholders

Step 5: Verification
Remediation Claimed → Trigger Rescan → Compare Results
→ If Fixed: Auto-Close Ticket | If Not Fixed: Reopen with Evidence

Step 6: Reporting
Aggregate Metrics → Tableau Data Source → Dashboard Refresh
→ Executive View (trends, SLA compliance, risk posture)

Key Design Decisions

  • Idempotency: All API calls and ticket creation are idempotent to handle retries safely
  • Human-in-the-Loop: LLM provides recommendations; analysts validate before ticket creation
  • Graceful Degradation: If CMDB lookup fails, ticket still created with manual review flag
  • Audit Trail: All decisions logged with timestamps, user IDs, and reasoning
  • Rate Limiting: Respect API rate limits with exponential backoff
  • Scalability: Horizontal scaling via Kubernetes for high-volume processing
